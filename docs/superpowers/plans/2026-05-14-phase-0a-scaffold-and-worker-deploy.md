# Phase 0a: Scaffold Next.js + Tailwind + shadcn on Cloudflare Workers — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up a new Next.js (App Router) + Tailwind v4 + shadcn/ui project at `next/` inside the existing repo, deployed to Cloudflare Workers via `@opennextjs/cloudflare` with the existing `SUBSCRIBERS` KV namespace bound. End state: a preview URL serves a "hello CampWatch" page proving the toolchain works end-to-end. The existing CRA app continues to serve production traffic untouched.

**Architecture:** Two Workers coexist in this Cloudflare account during the migration. The current CRA app deploys to the `campsites-finder` Worker via the existing `.github/workflows/deploy.yml` workflow. The new Next.js app lives at `next/` with its own `wrangler.jsonc`, `package.json`, and GitHub Actions workflow, and deploys to a separate `campwatch` Worker via `@opennextjs/cloudflare` (which targets Workers Static Assets). Both Workers bind the same `SUBSCRIBERS` KV namespace. Cloudflare Pages is intentionally avoided — it's in maintenance and the supported path for Next.js on Cloudflare is now Workers. Production traffic moves to the new Worker only at the end of Phase 0d.

**Tech Stack:** Next.js 15 (App Router, TypeScript), Tailwind CSS v4, shadcn/ui, lucide-react, next-themes, `@opennextjs/cloudflare`, Wrangler, pnpm (the Next.js scaffold's default; lighter lockfile than npm).

### Task B4: GitHub Actions workflow that deploys the Next.js Worker on every push

During Phase 0a–0c the `campwatch` Worker has no production traffic, so the workflow can safely overwrite it from any branch. After Phase 0d cutover, this workflow gets tightened to deploy only from `main` (and use `wrangler versions upload` for non-main previews so they don't clobber production). The existing CRA `deploy.yml` is untouched; the old `campsites-finder` Worker keeps shipping until we retire it.

**Files:**
- Create: `.github/workflows/deploy-next.yml`

- [ ] **Step 1: Create the workflow file**

```yaml
name: Deploy Next.js to Cloudflare Workers

on:
    push:
        branches: ["**"]
        paths:
            - "next/**"
            - ".github/workflows/deploy-next.yml"
    workflow_dispatch: {}

jobs:
    deploy:
        runs-on: ubuntu-latest
        defaults:
            run:
                working-directory: next
        steps:
            - uses: actions/checkout@v5

            - uses: pnpm/action-setup@v4
              with:
                  version: 9

            - uses: actions/setup-node@v5
              with:
                  node-version: 22
                  cache: pnpm
                  cache-dependency-path: next/pnpm-lock.yaml

            - name: Install
              run: pnpm install --frozen-lockfile

            - name: Build for Cloudflare Workers
              run: pnpm run cf:build

            - name: Deploy Worker
              uses: cloudflare/wrangler-action@v3
              with:
                  apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
                  accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
                  workingDirectory: next
                  command: deploy
```

- [ ] **Step 2: Confirm the required GitHub secrets already exist**

Visit `https://github.com/robertsmikej/campsites-react/settings/secrets/actions`. Expected: `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` are both present (they're used by the existing CRA workflow).

If either is missing, add it before continuing. The API token needs `Workers Scripts:Edit` and `Workers KV Storage:Edit` permissions for the relevant account (the existing token already covers these — verify rather than re-issue).
